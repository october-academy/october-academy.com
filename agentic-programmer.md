# Agentic Programmer

> '우리의 전문 분야는 특화되지 않는 것이다'

---

## 1. 개요

### Agentic Programmer란

Agentic Programmer는 **문제 정의 능력**과 **실행에 대한 책임**을 동시에 갖춘 프로그래머입니다.

AI가 코드를 작성하는 시대에, 우리의 차별화는 "무엇을 만들지"가 아닌 **"무엇을 풀지 결정하고, 그 결과에 책임지는 것"**에 있습니다. 코드를 쓰는 것은 AI도 할 수 있습니다. 하지만 불확실한 상황에서 문제를 정의하고, 실행하고, 결과에 책임지는 것은 인간의 영역입니다.

이것은 경력의 문제가 아닙니다. **마인드셋의 문제**입니다. 1년차도 Agentic할 수 있고, 10년차도 그렇지 않을 수 있습니다.

### General Purpose Programmer

Agentic Programmer는 CrossFit의 GPP(General Physical Preparation)를 프로그래밍에 적용한 개념입니다.

넷플릭스 예능 '피지컬 100'을 보면, 수많은 엘리트 운동 선수들이 등장합니다. 그런데 기발한 도전 과제와 예측 불가능한 미션이 주어질 때면, 크로스핏 선수들이 대체로 높은 수행력을 보여줍니다.

크로스핏 선수들은 마라톤 선수만큼 빠르지는 않지만 **충분히** 빠르고, 역도 선수만큼 무겁지는 않지만 **충분히** 강합니다.

우리가 추구하는 Agentic Programmer도 비슷합니다. 특정 도메인에서 최고가 되기보다는, **대부분의 영역에서 충분히 뛰어난 엔지니어**가 되는 것을 추구합니다.

단순히 여러 기술을 아는 제너럴리스트와는 다릅니다. GPP는 **문제 유형 기반**으로 충분한 깊이를 갖춥니다. 넓지만 얕은 것이 아니라, **필요한 곳에서 필요한 깊이**를 확보합니다.

---

## 2. 목적

### 왜 Agentic해야 하는가

우리는 커리어, 재정적 보상, 명예를 얻기 위해서 좋은 직장에 갑니다. 하지만 실제로 그런 곳에 가면 당신이 얻게 되는 것은 **황금수갑(Golden Handcuffs)** 뿐입니다.

아마 당신도 사람들이 좋아하는 직장에 가본 경험이 있다면 제 말이 무슨 의미인지 이해할 수 있을 겁니다.

당신이 아는 최고의 프로그래머들은 아마도 어떤 유명한 회사 "출신"(MAANG)일 것입니다. 현재는 그곳에 속해 있지 않겠죠. 그들은 **자유로운 삶을 선택**했습니다. 회사 이름에 기대지 않습니다.

**좋은 프로그래머는 여러분을 자유롭게 합니다.**
- 원치 않는 지하철
- 라이프사이클에 맞지 않는 9 to 6
- 취향 맞지 않는 동료들
- 내가 무얼 하는지도 모르는 boss

이 모든 것으로부터 벗어날 **선택권**을 갖게 됩니다.

### AI 시대의 핵심 위협

코딩 자동화로 인해 프로그래머의 역할이 축소되고 있습니다. 코드를 작성하는 것만으로는 더 이상 차별화되지 않습니다.

**생존 전략**: 불확실한 문제를 정의하고 해결하는 능력으로 차별화하는 것입니다. AI가 대체할 수 없는 영역에서 가치를 만들어야 합니다.

## 3. 대상

이 가이드는 모든 프로그래머를 위한 것이 아닙니다.

### 이런 성향의 사람들을 위한 가이드입니다

- **자율성을 중시하는 사람**: 누군가가 시키는 일보다 스스로 정의한 문제를 푸는 것을 선호
- **불확실성을 감내하는 사람**: 정해진 길보다 탐험을 선호하고, 실패 가능성을 수용할 수 있음
- **불편함을 감수하는 사람**: 성장을 위해 익숙하지 않은 영역에 뛰어들 준비가 됨

만약 안정적인 대기업 커리어를 원하고, 전문 분야 하나를 깊게 파고 싶다면, 이 가이드가 맞지 않을 수 있습니다. 그것도 훌륭한 선택입니다.

## 4. 원칙

CrossFit의 3대 원칙을 프로그래밍에 적용합니다.

### 1) Constantly Varied → 문제 유형의 지속적 변화

CrossFit에서는 매일 다른 운동을 합니다. 똑같은 루틴의 반복이 아닙니다.

Agentic Programmer도 마찬가지입니다. **매일 다른 유형의 문제를 풉니다.** 오늘은 프론트엔드, 내일은 API 설계, 모레는 시스템 아키텍처. 한 영역에 갇히지 않습니다.

### 2) High Intensity → 복합적 강도

CrossFit의 고강도는 단순히 "힘들다"가 아닙니다. 시간 제한, 무게, 복잡한 동작이 결합된 **복합적 강도**입니다.

프로그래밍에서의 고강도는:
- **시간 압박**: 제한된 시간 내에 동작하는 것을 만듦
- **복잡도**: 여러 기술이 결합된 문제
- **새로움**: 익숙하지 않은 기술 스택을 사용

이 세 가지가 결합될 때 진정한 성장이 일어납니다.

### 3) Functional Movement → 실제 프로젝트 기반

CrossFit은 기계 위에서 고립된 근육을 훈련하지 않습니다. 실제 삶에서 사용하는 **기능적 움직임**을 훈련합니다.

프로그래밍도 마찬가지입니다. 알고리즘 문제만 푸는 것이 아닙니다. **실제로 동작하는 무언가를 만듭니다.** End-to-end로 작동하는 것. 누군가가 실제로 사용할 수 있는 것.

## 5. 방법론

### 4가지 문제 유형

GPP가 다양한 신체 능력을 훈련하듯, Agentic Programmer는 4가지 핵심 문제 유형을 훈련합니다:

| 영역 | 설명 | 예시 |
|-----|------|-----|
| **UI/프론트엔드** | 사용자와 직접 상호작용하는 인터페이스 | 반응형 레이아웃, 애니메이션, 상태 관리 |
| **API/백엔드** | 데이터 처리와 비즈니스 로직 | REST/GraphQL 설계, 인증, 데이터베이스 |
| **시스템 설계** | 확장성과 아키텍처 | 마이크로서비스, 캐싱, 메시지 큐 |
| **알고리즘/데이터구조** | 효율적인 문제 해결 | 최적화, 복잡도 분석, 자료구조 선택 |

우리는 각 영역에서 전문가 수준이 되려는 것이 아닙니다. 각 영역에서 **충분히 기능할 수 있는 수준**을 목표로 합니다.

### Programming WOD (Workout of the Day)

CrossFit에서 매일 WOD를 수행하듯, 프로그래밍에서도 일일 훈련 형식을 정의합니다.

#### AMRAP 스타일 (As Many Rounds As Possible)
**30분 내 최대한 구현하기**

예시:
- 30분 동안 TODO 앱의 핵심 기능 최대한 구현
- 30분 동안 API 엔드포인트 최대한 구축

측정: 완성한 기능의 수, 동작하는 코드의 양

#### For Time 스타일
**특정 과제를 얼마나 빨리 끝내는가**

예시:
- 로그인/회원가입 기능 완성까지 걸리는 시간
- CRUD API 구축 시간 측정

측정: 완료까지 걸린 시간

#### EMOM 스타일 (Every Minute On the Minute)
**매일 같은 유형의 작은 과제**

예시:
- 매일 하나의 알고리즘 문제 풀기
- 매일 하나의 UI 컴포넌트 만들기

측정: 연속 수행 일수, 품질의 변화

### Benchmark: 클론 프로젝트

CrossFit에는 "Fran", "Murph" 같은 Benchmark WOD가 있습니다. 자신의 발전을 측정하는 기준점이 됩니다.

프로그래밍의 Benchmark는 **유명 서비스의 핵심 기능 클론**입니다:

| 레벨 | 클론 대상 | 핵심 구현 요소 |
|-----|---------|--------------|
| 입문 | Todo App | CRUD, 로컬 상태 관리 |
| 초급 | Twitter 클론 | 피드, 팔로우, 실시간 업데이트 |
| 중급 | Slack 클론 | 실시간 메시징, 채널, 검색 |
| 고급 | Notion 클론 | 블록 에디터, 협업, 동기화 |

같은 클론 프로젝트를 6개월 후에 다시 해보세요. 시간이 얼마나 단축되었는지가 당신의 성장입니다.

### 기본기 (Foundational Movement)

CrossFit에서 스쿼트, 데드리프트, 프레스가 모든 움직임의 기초이듯, 프로그래밍의 기본기는:

**"동작하는 무언가 만들기"**

- 실제로 실행되는 코드
- End-to-end로 작동하는 기능
- 누군가에게 보여줄 수 있는 결과물

아무리 복잡한 이론을 알아도, 동작하는 것을 만들지 못하면 의미가 없습니다.

## 6. 적용

### 의도적 훈련 vs 그냥 코딩

많은 프로그래머들이 "일하면서 늘지 않나요?"라고 묻습니다.

차이는 **측정과 기록**입니다.

그냥 운동하는 것과 훈련의 차이는, 기록을 하느냐 안 하느냐입니다. 10년 동안 헬스장에 다녀도 기록 없이 느낌대로 하면 성장이 정체됩니다. 하지만 매 세트, 매 무게를 기록하면 점진적으로 발전합니다.

프로그래밍도 마찬가지입니다.

### 4가지 측정 항목

#### 1) 시간 기록
훈련에 투자한 시간을 기록합니다.
- 오늘 몇 시간 코딩했는가?
- 그 중 업무 외 훈련 시간은?
- 어떤 유형의 문제에 시간을 썼는가?

#### 2) 결과물 기록
만들어낸 것들의 목록을 유지합니다.
- 완성한 프로젝트
- 작성한 기능
- 해결한 문제

#### 3) 회고 기록
매 훈련 후 짧게 기록합니다.
- 오늘 배운 것
- 어려웠던 것
- 다음에 개선할 점

#### 4) 외부 피드백
혼자만의 평가는 맹점이 있습니다.
- 코드 리뷰
- 블로그에 대한 댓글
- 프로젝트 사용자의 반응

### 일일 루틴

**활동 기반**으로 루틴을 설정합니다. "언제"가 아니라 "무엇을"에 집중합니다.

업무만으로는 부족합니다. 업무는 대부분 익숙한 영역의 반복입니다. **별도의 훈련 시간**이 필요합니다.

권장하는 최소 루틴:
- 매일 30분 이상의 의도적 훈련
- 주 1회 이상 새로운 문제 유형 시도
- 주 1회 이상 배운 것 공유

### 공유의 중요성

> **Luck = Doing × Telling**

실력을 쌓는 것(Doing)만으로는 부족합니다. 그것을 알리는 것(Telling)이 곱해져야 기회(Luck)가 됩니다.

- 블로그에 배운 것 정리
- GitHub에 프로젝트 공개
- SNS에 진행 상황 공유
- 커뮤니티에서 질문/답변

숨어서 잘하는 것은 아무도 모릅니다. **보이는 곳에서 성장하세요.**

### 첫 발

지금 당장 시작할 수 있는 가장 작은 단계:

**오늘 배운 것 기록하기**

복잡한 시스템이나 완벽한 계획이 필요 없습니다. 노트 앱이든, 메모장이든, 어디든 좋습니다.

오늘 코딩하면서 새로 알게 된 것 하나만 적어보세요. 내일도, 모레도. 한 달이 지나면 30개의 배움이 쌓입니다.

이것이 Agentic Programmer로 가는 첫 발입니다.

## 7. 결과

### 기대되는 변화

Agentic Programmer로 훈련하면:

**단기 (3-6개월)**
- 새로운 기술 습득 속도 향상
- 문제 정의 능력 향상
- 자신감 있게 불확실한 문제에 접근

**중기 (1-2년)**
- 어떤 프로젝트든 기여할 수 있는 범용성
- 기술적 의사결정에서 신뢰받는 위치
- 이직 시 선택의 폭 확대

**장기 (3년+)**
- 회사에 의존하지 않는 커리어 자율성
- 창업, 프리랜서, 어떤 형태든 선택 가능
- 기술 변화에도 흔들리지 않는 적응력

### 성공 지표

#### 커리어 지표
- 연봉 상승
- 더 좋은 직책
- 이직 기회의 증가
- 면접 합격률

#### 영향력 지표
- 팔로워 수
- 오픈소스 기여
- 블로그 조회수
- 강연/출판 기회

숫자가 전부는 아니지만, **측정할 수 있어야 개선할 수 있습니다.**

### Agentic한 사람들의 패턴

제가 관찰한 Agentic한 프로그래머들의 공통점:

1. **자기 주도성**: 시키지 않아도 문제를 찾아서 해결
2. **불편함 감수**: 익숙한 것에 안주하지 않음
3. **공개적 성장**: 배우는 과정을 숨기지 않음
4. **실행 우선**: 완벽한 계획보다 일단 시작

## 8. 한계

### GPP의 본질적 한계

솔직히 말해야 합니다.

General Purpose Programmer는 **깊은 전문성이 요구되는 상황에서 약합니다.**

- 데이터베이스 내부 구조를 설계해야 할 때
- 컴파일러를 처음부터 만들어야 할 때
- 특정 도메인의 최적화가 필요할 때

이런 상황에서는 10년 동안 그것만 파온 전문가를 이길 수 없습니다.

### 대응 전략

한계를 인정하고, **적합한 상황을 선택**합니다.

Agentic Programmer가 빛나는 상황:
- 문제가 정의되지 않은 초기 단계
- 여러 기술이 결합된 프로젝트
- 빠른 프로토타이핑이 필요할 때
- 스타트업처럼 한 사람이 여러 역할을 해야 할 때

Agentic Programmer가 약한 상황:
- 깊은 전문성이 명확히 요구될 때
- 대규모 조직에서 좁은 역할이 정해질 때

자신의 강점이 발휘되는 환경을 **의도적으로 선택**하는 것도 Agentic함의 일부입니다.

---

## 시작하세요

Agentic Programmer가 되는 것은 특별한 재능이 필요하지 않습니다.

필요한 것은:
- 매일 조금씩 훈련하는 것
- 기록하는 것
- 공유하는 것

**오늘 배운 것을 기록하는 것**부터 시작하세요.

그것이 당신의 첫 번째 WOD입니다.
